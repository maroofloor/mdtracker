## 2. 발견된 문제 (영상 근거)

### P1. 항복 DEFEAT를 놓칠 수 있는 폴링 루프 구조

`recognize_server_error`가 **매 주기, 모든 상태에서** 실행된다. server_error ROI(화면 중앙
0.43~0.58)는 듀얼 중 카드/이펙트로 밝은 픽셀이 흔해 게이트를 통과 → tesseract 한글 OCR이
0.25초마다 돈다. OCR 1회 200~500ms이므로 실효 주기가 0.5초 이상으로 늘어나,
**0.75초짜리 항복 DEFEAT 창을 통째로 놓칠 수 있다.**

수정:
- 루프에서 **상태별 인식(특히 결과 템플릿 매칭 = 저비용)을 먼저** 수행하고 서버 오류 검사는 뒤로.
- 서버 오류 OCR은 **N주기마다 1회**(예: 8주기 ≈ 2초)로 스로틀. 팝업은 사용자가 닫을 때까지
  지속되므로 감지율 손실 없음.

### P2. 듀얼 중 텍스트 배너가 toss ROI와 겹침 → 오탐 위험

듀얼 중 효과 발동/상대 행동 안내 배너("…효과를 발동할 수 있습니다", "상대가 …하고 있습니다" 류)가
coin/toss ROI와 같은 화면 띠(y 0.60~0.72)에 출현하는 것을 231728 영상에서 확인.
현재 토스 패배 판정 규칙 `"상대" + "있습니다"`는 이런 배너와 충돌할 수 있다
(예: "상대가 효과 대상을 선택하고 있습니다").

수정 — 키워드 강화. 실제 토스 화면 문구는 선공·후공이 **둘 다** 들어 있다:
- 토스 승: compact에 `선공` AND `후공` AND `주세요`
- 토스 패: compact에 `선공` AND `후공` AND `상대`
- 선후공 확정: `당신이` AND (`선공` XOR `후공`) — 기존 유지

이 강화는 R1의 복구 신호를 안전하게 만드는 전제 조건이기도 하다.

트레이드오프: AND 조건 강화는 OCR이 `선공`·`후공`을 **둘 다 정확히 읽어야** 인정하므로
미감지율이 오를 수 있다. IDLE에서 토스는 유일한 듀얼 진입 신호라 미감지 = 그 판 기록 누락.
다중 임계 시도 + 8~12초(패)/2~4초(승) 노출이면 실측상 충분할 가능성이 높지만,
**강화 전후 영상 9개에서 토스 감지율 회귀 확인을 검증 항목에 포함**한다.

## 3. 정체 복구 메커니즘 (타임아웃 대체)

게임 길이(30초~1시간)에 의존하는 PLAYING 타임아웃은 쓰지 않는다.
대신 "현재 상태와 양립 불가능한 이벤트"를 복구 신호로 쓴다. 우선순위 순:

### R1. 새 코인토스 = 자가 복구 (필수) — **PLAYING 한정**

**PLAYING 상태에서만** 토스 인식(P2 강화 규칙)을 함께 돌린다.
새 토스 감지 = 이전 듀얼의 결과를 놓친 것이 확정이므로:

1. 이전 듀얼을 결과 미상(`result='unknown'`)으로 발행 → §6의 미상 결과 파이프라인이 처리
2. 새 토스 값으로 COIN 재진입

**COIN 상태에는 R1을 적용하지 않는다.** 토스 문구는 8~12초 노출되는데 COIN 진입은
토스 감지 *직후*라, COIN에서 토스 인식을 돌리면 같은 화면의 잔상을 "새 토스"로 재감지해
오발행 + 재진입을 반복한다. COIN 정체는 R3(타임아웃)가 전담한다.
PLAYING 진입은 "당신이 선공/후공입니다" 확정 후라 토스 잔상이 없어 안전하다.

**토스 OCR 스로틀 (필수).** PLAYING에서 토스 OCR을 매주기 돌리면 P2에서 지적한
"듀얼 중 텍스트 배너가 coin ROI 게이트를 통과"하는 상황마다 tesseract가 돌아,
P1에서 제거한 실효 주기 증가 문제를 재도입한다(0.75초 항복 DEFEAT 창 누락 위험 부활).
토스 문구는 8~12초 노출이므로 **N주기마다 1회(2~3초)** 스로틀해도 감지율 손실 없음.
PLAYING 루프 순서: 결과 템플릿 매칭(저비용, 매주기) → 토스 OCR(스로틀) → 서버 오류 OCR(스로틀).

근거: 토스 패배 문구는 8~12초 노출(가장 긴 신호)이라 감지 확실. 매치 사이 간격은
보상 화면 ~15초 + 로비 체류뿐이므로 복구 지연은 "다음 판 시작까지"로 자연 한정.
주의: `result_cooldown`을 토스 감지에도 적용해 결과 직후 잔상 오인 방지
(`_cooldown_until`은 `_reset_state()`에서 지워지지 않으므로 그대로 재사용 가능).

### R2. 게임 창 소멸 (필수, 공짜)

`capture_screen()`이 None(창 미발견)을 연속 N회(예: 40회 ≈ 10초) 반환하고 상태가
IDLE이 아니면 → R1과 동일 처리 후 IDLE. 게임 종료/크래시 시 유일하게 동작하는 경로.

경계 조건 두 가지:
- **가림은 소멸이 아니다.** 현재 캡처는 창 좌표 기준 화면 영역 grab이라, 다른 창이 게임을
  덮어도 창 탐색(EnumWindows)은 성공 → None이 아니므로 R2는 발동하지 않는다(오작동 없음).
  대신 가린 창의 픽셀을 읽게 되는 문제는 §4의 PrintWindow 전환으로 해결한다.
- **최소화를 소멸로 오인하지 않는다.** 최소화된 창은 GetClientRect가 0 크기를 줄 수 있어
  None 반환 → R2가 "창 소멸"로 오인할 수 있다. `win32gui.IsIconic(hwnd)`이면
  "캡처 불가지만 듀얼 진행 중일 수 있음"으로 취급해 **R2 카운터를 올리지 않고 상태 유지**,
  복원 시 캡처 재개. 최소화 중 결과 화면이 지나간 경우는 R1이 회수한다.

**캡처 반환 계약 확장 필요.** 현재 `capture_screen()`의 None은 "창 없음"과
"GetClientRect 0 크기(최소화)"를 구분하지 않는다(`engine.py`의 `cw <= 0 or ch <= 0`
경로). IsIconic 가드를 폴러에서 쓰려면 None/이미지 외에 제3의 신호가 필요하다 —
예: `engine.last_capture_status` 속성('ok' | 'not_found' | 'minimized') 또는
상태를 함께 반환하는 별도 메서드. 즉 R2는 engine.py와 poller.py 양쪽을 건드리며,
§4의 "반환 계약 동일" 문구는 캡처 *이미지* 계약에만 해당한다.

### R3. COIN 한정 짧은 타임아웃 (필수, 한 줄)

COIN 구간(토스 확인 → 선후공 확정)은 영상 실측 15~25초로 상한이 있는 유일한 구간.
120초 초과 시 토스 정보만 버리고 IDLE 복귀(발행 없음 — 듀얼 데이터가 토스뿐이라 review 가치 없음).
서버오류 영상(090952)처럼 토스 후 CONNECTING에서 멈추고 팝업 OCR마저 실패하는 경우의 안전망.
`_state_entered_at` 필드가 이미 있어 구현이 한 줄이다.

### R4. 듀얼 HUD 하트비트 (선택, 후순위)

"듀얼 중에는 LP 표시가 항상 있다"를 역이용해 PLAYING에서 HUD 부재가 지속되면 듀얼 종료로
간주하는 방안도 검증했으나, 영상에서 다음 가림 요인이 확인됐다:
- 카드 상세 패널이 왼쪽 열 전체(내 LP 포함)를 장시간 가림 (090828)
- 컷인/공격 연출이 화면 전체를 ~4초 이상 가림 (232325에서 최대 29프레임 ≈ 7초)
- 상대 LP 표기는 내 LP와 스타일이 달라 단일 휴리스틱 불가

→ 다중 앵커 + 20~30초 부재 임계가 필요해 복잡도 대비 이득이 작다.
R1~R3만으로 모든 정체 시나리오(결과 미감지·크래시·연결 끊김)가 커버되므로 보류 권장.

## 4. 캡처 방식 전환: 화면 영역 grab → PrintWindow (권장)

가림 문제의 근본 해결책. 화면 좌표를 mss로 찍는 대신, DWM이 보관한 창 자체의 렌더링
표면을 복사하는 `PrintWindow(hwnd, dc, PW_RENDERFULLCONTENT)`로 전환한다.

### 대응 범위 (정확한 한계 포함)

| 시나리오 | 화면 영역 grab (현재) | PrintWindow |
|---|---|---|
| 다른 창이 게임을 가림 | 가린 창 픽셀을 읽음 (오인식·결과 누락) | **게임 화면 정상 캡처** |
| 창모드 + 멀티태스킹 (상대 턴에 브라우저 등을 위에 띄움) | 추적 중단·오인식 | **추적 유지** — 가장 체감 큰 개선 |
| 창이 모니터 가장자리 밖으로 일부 벗어남 | 잘린/깨진 캡처 | **전체 내용 캡처** |
| 전체화면 독점 모드 | 정상 (가릴 창이 없음) | 검은 프레임 위험 → **화면 grab 폴백으로 손해 없음** |
| 최소화 | None 반환 → R2 오인 위험 | 실패하거나 깨진 프레임 — **해결 불가** |
| 창 종료/크래시 | None → R2 정상 동작 | 동일 |

최소화는 어떤 캡처 기술로도 보장이 안 된다: Unity 게임은 최소화 시 렌더링 자체를 멈출 수
있어(Run In Background 설정에 따라) WGC 같은 상위 기술도 멈춘 프레임밖에 못 얻는다.
→ 최소화는 캡처가 아니라 R2의 IsIconic 가드 + R1 복구로 처리한다 (역할 분담).

### 구현 범위 — engine.py 1개 파일, 새 의존성 없음

- `capture_screen()` 내부 교체 + 헬퍼 1개 (약 50~70줄). 창 탐색(EnumWindows)은 재사용.
  win32ui/win32con은 이미 의존 중인 pywin32에 포함 — requirements.txt 변경 없음.
- `PW_CLIENTONLY | PW_RENDERFULLCONTENT` 플래그로 타이틀바를 제외한 클라이언트 영역만
  캡처해야 비율 기반 ROI가 그대로 호환된다.
- 캡처 이미지 계약 동일(PIL 이미지 또는 None) — PrintWindow 전환 자체는 poller/UI/테스트
  변경 없음. 단, R2의 최소화 구분용 캡처 상태 신호(위 §3 R2 참조)는 별도로 추가된다.
- **검은 프레임 폴백**: 일부 GPU/전체화면 독점 모드에서 PrintWindow가 전부 검정 프레임을
  반환하는 사례가 있다. 캡처 결과가 사실상 전흑이면 기존 화면 영역 grab으로 자동 폴백한다.
  (mss는 모니터 모드용으로도 필요하므로 그대로 유지.)
- 리스크는 실 PC 검증 1건뿐: Master Duel(Unity)은 Win10+ 창모드/테두리없음에서
  PW_RENDERFULLCONTENT가 일반적으로 동작하지만 사용 환경에서 1회 확인 필요.

### 가림 관련 잔여 리스크 (전환 전까지)

PrintWindow 전환 전에는, 가린 창에 "선공/후공" 류 텍스트가 해당 ROI 위치에 보이면
(예: 마스터듀얼 방송 시청, 관련 문서 열람) 강화 키워드로도 오탐이 가능하다.
needs_review 발행이라 피해는 확인 팝업 1회 수준이며, 전환 후에는 소멸한다.

## 5. 구현 시 함께 정리할 것

- `poller.py` 독스트링 10행 "playing_timeout … 링 버퍼 소급 검사"는 미구현 설명 — 본 설계로 교체
- 모든 복구 경로에서 `state_changed` 시그널 발행 유지 (OCR 패널 가시성)
- 단위 테스트: `_handle_*`에 가짜 인식 결과를 주입하는 테스트를 `tests/`에 추가 (tesseract 불필요).
  R1 잔상 가드(COIN 미적용)·R2 카운터/IsIconic·R3 타임아웃 케이스 포함
- P2 키워드 강화 전후 영상 9개 토스 감지율 회귀 확인 (§2 P2 참조)

## 6. 결과 미상(`unknown`) 파이프라인 — R1·R2의 전제 조건, 별도 단계

R1·R2는 "결과 미상 발행"에 의존하는데, **현재 파이프라인은 이를 받을 수 없다.**
코드 대조에서 확인된 충돌:

| 지점 | 현재 코드 | 충돌 |
|---|---|---|
| DB 스키마 | `result TEXT NOT NULL` (db.py) | `result=None`이면 INSERT 제약 위반 예외 → 기록 유실 |
| 도메인 모델 | `Match.result: str` — 'win'/'loss'/'draw' (models.py) | None 허용 안 됨 |
| 저장 흐름 | `_on_ocr_detected`가 팝업 확인 없이 **즉시 DB 저장** (record_view.py) | "팝업에서 승/패 입력 후 저장" 흐름이 존재하지 않음 |
| 확인 패널 | `ocr_panel`은 상대 덱 교정 전용 | 승/패 입력 UI 없음 |
| 상태 문구 | `RESULT_LABELS.get(m.result, m.result)` | result=None이면 "None 기록됨" 깨진 문구 |
| coin 기본값 | `coin_result=res.coin_result or "first"` | 미상 발행은 coin_result=None일 수 있어 조용히 'first'로 오염 → 선후공 통계 왜곡 |
| review 표시 | 행 강조가 `opponent_deck == 미정` 기준 | 결과 미상 행은 덱만 채우면 표시가 사라짐 |

따라서 R1·R2의 폴러 수정과 별개로 다음을 **선행 단계**로 구현한다:

1. **허용값 결정**: `result`에 `'unknown'` 추가 (NULL 허용 마이그레이션보다 단순 —
   NOT NULL 유지, models.py 허용값 상수·labels.py 라벨("미상")만 추가).
   stats.py는 win/loss/draw 분기만 집계하므로 unknown은 자연히 승률 분모에서 빠진다 —
   의도된 동작임을 stats.py 독스트링에 명시.
2. **ocr_panel 확장**: 승/패/무 선택 UI 추가. 미상 발행 시 패널을 항상 표시하고
   (현재는 상대 덱 미입력 시만), 사용자가 결과를 고르면 `result` 갱신 + `confirmed=True`.
3. **coin 기본값 제거**: `coin_result or "first"` 폴백 삭제. 미상 발행에서 coin_result가
   None이면 그대로 저장(스키마의 `coin_result NOT NULL`도 함께 재검토)하거나
   needs_review로만 처리 — 'first' 추정 금지 (CLAUDE.md "OCR은 추정하지 않는다" 원칙).
4. **review 행 식별**: `result == 'unknown'`도 행 강조 조건에 추가.

이 단계가 끝나기 전까지 R1·R2는 미상 발행 대신 "발행 없이 IDLE 복귀"로 동작시킬 수 있다
(복구는 되지만 놓친 듀얼은 기록되지 않음 — 임시 안전 모드).
