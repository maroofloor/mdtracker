2. 브랜치 전략

GitHub Flow를 기반으로 단순하게 운영합니다.

main 항상 배포 가능한 안정 버전. 직접 커밋 금지.
└─ develop 다음 릴리스 통합 브랜치. 피처 브랜치 병합 대상.
├─ feat/ocr-window-mode 기능 개발
├─ feat/export-excel 기능 개발
└─ fix/coin-toss-detection 버그 수정

브랜치 명명 규칙

접두사용도예시
feat/ 신규 기능 feat/dashboard-trend
fix/ 버그 수정 fix/ocr-false-positive
chore/ 빌드·설정·의존성 chore/update-pyside6
docs/ 문서 수정 docs/update-readme
refactor/ 기능 변경 없는 코드 정리 refactor/stats-module

3. 커밋 메시지 규칙

Conventional Commits 형식을 사용합니다.

형식
<type>(<scope>): <subject>
예시
feat(ocr): 창 모드 게임 창 자동 감지 추가
fix(poller): 코인토스 판정 오탐 수정
chore(deps): PySide6 6.7로 업그레이드
docs: README 설치 가이드 보완

type 종류

type설명
feat사용자에게 노출되는 새 기능
fix버그 수정
chore빌드, 의존성, 설정 변경
docs문서만 변경
refactor기능 변경 없는 리팩토링
test테스트 추가·수정
perf성능 개선

4. 버전 관리 (Semantic Versioning)

버전 형식: MAJOR.MINOR.PATCH  예: v1.2.0

구분변경 시점예시
MAJOR DB 스키마 파괴적 변경, UI 전면 개편 v2.0.0
MINOR 신규 기능 추가 (하위 호환 유지) v1.1.0
PATCH 버그 수정, 성능 개선 v1.0.1

태그 & 릴리스 절차

develop → main PR 머지 후 실행
git tag -a v1.0.0 -m "Release v1.0.0"
git push origin v1.0.0
→ GitHub Releases에 CHANGELOG 작성 후 게시
